Archgate-regler er TypeScript-filer som eksporterer et vanlig objekt typet med `satisfies RuleSet`. Hver regel mottar en `RuleContext` med verktﾃｸy for ﾃ･ sﾃｸke i filer, lese innhold og rapportere brudd.

## RuleSet

```typescript
/// <reference path="../rules.d.ts" />

export default {
  rules: {
    "my-rule-id": {
      description: "Human-readable description of what this rule checks",
      severity: "error", // optional, defaults to "error"
      async check(ctx) {
        // Rule logic here
      },
    },
  },
} satisfies RuleSet;
```

En regelfil eksporterer som standard et vanlig objekt med en `rules`-post indeksert etter regel-ID. Nﾃｸklene blir regel-ID-ene som vises i sjekkutdata og bruddrapporter. Annotasjonen `satisfies RuleSet` gir typekontroll uten ﾃ･ pakke inn i et funksjonskall.

```typescript
type RuleSet = { rules: Record<string, RuleConfig> };
```

## RuleConfig

Hver regel i posten mﾃ･ samsvare med `RuleConfig`-grensesnittet.

```typescript
interface RuleConfig {
  description: string;
  severity?: Severity;
  check: (ctx: RuleContext) => Promise<void>;
}
```

| Felt          | Type                                  | Pﾃ･krevd | Beskrivelse                                                |
| ------------- | ------------------------------------- | ------- | ---------------------------------------------------------- |
| `description` | `string`                              | Ja      | Lesbar beskrivelse som vises i sjekkutdata                 |
| `severity`    | `Severity`                            | Nei     | Standard alvorlighetsgrad for brudd. Standard er `"error"` |
| `check`       | `(ctx: RuleContext) => Promise<void>` | Ja      | Asynkron funksjon som inneholder regellogikken             |

## RuleContext

`check`-funksjonen mottar et `RuleContext`-objekt med prosjekttilstand og hjelpemetoder.

```typescript
interface RuleContext {
  projectRoot: string;
  scopedFiles: string[];
  changedFiles: string[];
  glob(pattern: string): Promise<string[]>;
  grep(file: string, pattern: RegExp): Promise<GrepMatch[]>;
  grepFiles(pattern: RegExp, fileGlob: string): Promise<GrepMatch[]>;
  readFile(path: string): Promise<string>;
  fileAtBase(path: string): Promise<string | null>;
  readJSON(path: string): Promise<unknown>;
  ast(path: string, language: AstLanguage, opts?: AstOptions): Promise<AstNode>;
  report: RuleReport;
}
```

### Egenskaper

#### projectRoot

```typescript
projectRoot: string;
```

Absolutt sti til prosjektets rotkatalog (der `.archgate/` ligger).

#### scopedFiles

```typescript
scopedFiles: string[];
```

Filer som matcher ADR-ens `files`-globmﾃｸnstre fra frontmatter. Hvis ADR-en ikke har et `files`-felt, inneholder denne alle prosjektfiler. Bruk denne som den primﾃｦre fillisten for regelkontrollene dine.

#### changedFiles

```typescript
changedFiles: string[];
```

Filer som har blitt endret ifﾃｸlge git. Som standard fylles denne automatisk med branch-diffen mot den detekterte basisgrenen (f.eks. `origin/main`) pluss eventuelle ikke-committede endringer i arbeidstreet (stagede, ustagede og usporede ikke-ignorerte filer). Nﾃ･r `--staged` brukes, inneholder den kun stagede filer. Nﾃ･r `--base <ref>` brukes, inneholder den alle filer som er endret siden den referansen pluss ikke-committede endringer i arbeidstreet. Tom nﾃ･r basisdeteksjon feiler eller ingen endringer finnes. Bruk denne til ﾃ･ bygge regler for avhengigheter mellom filer (f.eks. "hvis fil A endret seg, mﾃ･ fil B ogsﾃ･ endres").

#### report

```typescript
report: RuleReport;
```

Rapporteringsgrensesnittet for ﾃ･ registrere brudd, advarsler og informasjonsmeldinger. Se [RuleReport](#rulereport) nedenfor.

### Metoder

#### glob

```typescript
glob(pattern: string): Promise<string[]>;
```

Finn filer som matcher et globmﾃｸnster relativt til prosjektroten. Returnerer en matrise med filstier. Filer som ignoreres av `.gitignore` er ekskludert som standard. Sett `respectGitignore: false` i ADR-ens frontmatter for ﾃ･ inkludere dem.

```typescript
const testFiles = await ctx.glob("tests/**/*.test.ts");
```

#### grep

```typescript
grep(file: string, pattern: RegExp): Promise<GrepMatch[]>;
```

Sﾃｸk i en enkelt fil etter linjer som matcher et regulﾃｦrt uttrykk. Returnerer en matrise med `GrepMatch`-objekter med filsti, linjenummer, kolonne og matchet innhold.

```typescript
const matches = await ctx.grep(file, /console\.error\(/);
```

#### grepFiles

```typescript
grepFiles(pattern: RegExp, fileGlob: string): Promise<GrepMatch[]>;
```

Sﾃｸk i flere filer som matcher et globmﾃｸnster etter linjer som matcher et regulﾃｦrt uttrykk. Kombinerer `glob` og `grep` i ett enkelt kall. Filer som ignoreres av `.gitignore` er ekskludert som standard. Sett `respectGitignore: false` i ADR-ens frontmatter for ﾃ･ inkludere dem.

```typescript
const matches = await ctx.grepFiles(/TODO:/i, "src/**/*.ts");
```

#### readFile

```typescript
readFile(path: string): Promise<string>;
```

Les innholdet i en fil som en streng. Stien er relativ til prosjektroten.

```typescript
const content = await ctx.readFile("src/config.ts");
```

#### fileAtBase

```typescript
fileAtBase(path: string): Promise<string | null>;
```

Les en fils kildekode ved den **sammenligningsgrunnlagsrevisjonen** -- fellespunktet (merge base) mellom `--base`-referansen og `HEAD`, den samme commiten `changedFiles` beregnes mot. Bruk den til ﾃ･ sammenligne arbeidstreet mot punktet der endringssettet delte seg fra basisen.

Returnerer `null` i de to "ingenting ﾃ･ sammenligne mot"-tilfellene, slik at en enkelt null-sjekk dekker begge:

- **Ingen basis er lﾃｸst** -- sjekken kjﾃｸrte uten `--base` (eller historikkene er urelaterte).
- **Filen fantes ikke ved basisen** -- en lagt-til fil.

```typescript
const before = await ctx.fileAtBase("data/schema.py");
if (before === null) {
  // Ingen basisversjon ﾃ･ sammenligne mot -- hopp over.
  return;
}
const after = await ctx.readFile("data/schema.py");
```

For en **strukturell** sammenligning (som ignorerer kommentarer og formatering), foretrekk `ast(path, language, { rev: "base" })` nedenfor.

#### readJSON

```typescript
readJSON(path: string): Promise<unknown>;
```

Les og parse en JSON-fil. Stien er relativ til prosjektroten. Returnerer den parsede verdien som `unknown` -- cast til forventet type i regelen din.

```typescript
const pkg = (await ctx.readJSON("package.json")) as {
  dependencies?: Record<string, string>;
};
```

#### ast

```typescript
ast(
  path: string,
  language: "typescript" | "javascript" | "python" | "ruby",
  opts?: AstOptions
): Promise<AstNode>;

interface AstOptions {
  rev?: "base";
  comments?: boolean;
}
```

Parse en kildefil til dens sprﾃ･knative AST. Stien er relativ til prosjektroten og gﾃ･r gjennom samme sandkasse som `readFile`. TypeScript og JavaScript parses i prosessen; Python og Ruby parses ved ﾃ･ starte systemtolkens egen AST-fasilitet fra standardbiblioteket som en underprosess. Formen pﾃ･ det returnerte treet varierer per sprﾃ･k -- se [AstNode](#astnode).

```typescript
const program = await ctx.ast("src/cli.ts", "typescript");
for (const node of program.body) {
  console.log(node.type);
}
```

##### Parsing av grunnrevisjonen (`{ rev: "base" }`)

Med `{ rev: "base" }` parser `ast()` filen ved den [sammenligningsgrunnlagsrevisjonen](#fileatbase) i stedet for arbeidstreet -- alt annet (returform, kastekontrakt) er identisk. Parse begge revisjonene for ﾃ･ spﾃｸrre "endret den kjﾃｸrbare strukturen seg?" Kommentarer er fravﾃｦrende fra ESTree- og Python-`ast`-formene, men det er ikke node**posisjonene**: `loc`- / `lineno`-feltene forskyves nﾃ･r en kommentar eller tom linje flytter linjene under seg. Sammenlign en **posisjonsfri projeksjon** -- fjern `loc` / `range` (ESTree) og `lineno` / `col_offset` (Python) fﾃｸr sammenligning -- slik at en ren kommentarendring leses som uendret. (Python-dokstrenger er strengnoder i treet, sﾃ･ en endret dokstreng er en reell endring; fjern dem ogsﾃ･ hvis dokumentasjonsendringer skal vﾃｦre nﾃｸytrale.)

```typescript
// Merk en endring bare nﾃ･r den kjﾃｸrbare strukturen faktisk endret seg.
// `structurallyEqual` sammenligner trﾃｦrne med posisjonsmetadata (loc/range,
// lineno/col_offset) fjernet -- se writing-rules-guiden for en konkret
// implementasjon.
for (const file of ctx.changedFiles.filter((f) => f.endsWith(".py"))) {
  // Hopp over filer uten basismotpart -- ast({ rev: "base" }) ville kastet.
  if ((await ctx.fileAtBase(file)) === null) continue;
  const before = await ctx.ast(file, "python", { rev: "base" });
  const after = await ctx.ast(file, "python");
  if (!structurallyEqual(before, after)) {
    ctx.report.violation({ message: `${file} changed behavior`, file });
  }
}
```

Bruk `fileAtBase()` fﾃｸrst nﾃ･r du trenger ﾃ･ oppdage tilfellene uten basis eller med lagt-til fil som ordinﾃｦr kontrollflyt -- `ast({ rev: "base" })` **kaster** for disse (se nedenfor).

##### Samle inn kommentarer (`{ comments: true }`)

Med `{ comments: true }` bﾃｦrer det returnerte treet en `comments`-matrise -- strukturert kommentardata for kommentarstyringsregler, i stedet for linje-for-linje regulﾃｦre uttrykk. Stﾃｸttet for alle fire sprﾃ･k. For `ruby` er det returnerte treet en matrise (`Ripper.sexp`-utdata), sﾃ･ `comments`-matrisen ligger pﾃ･ den som en ikke-indeksert egenskap.

```typescript
interface CommentToken {
  type: "line" | "block";
  value: string; // skilletegn (`//`, `/* */`, `#`) fjernet
  loc: {
    start: { line: number; column: number };
    end: { line: number; column: number };
  };
}
```

```typescript
const tree = await ctx.ast("src/api.ts", "typescript", { comments: true });
for (const comment of tree.comments ?? []) {
  const lines = comment.value.split("\n").length;
  if (lines > 10) {
    ctx.report.warning({
      message: "Comment block is too long -- link to an ADR instead",
      file: "src/api.ts",
      line: comment.loc.start.line,
    });
  }
}
```

Kommentarposisjoner er nﾃｸyaktige mot **originalkilden**, selv for TypeScript. Dette er en bevisst fordel fremfor treets egen `loc`, som er transpilert-relativ for TypeScript (se [AstNode](#astnode)): kommentarer skannes fra kildekoden fﾃｸr transpilering, sﾃ･ deres `loc` driver aldri. Python-kommentarer er alltid `type: "line"` (`#`) -- Python har ingen blokkommentarer, og `"""`-dokstrenger er strengeuttrykk i treet, ikke kommentarer. Ruby `#`-kommentarer er `type: "line"`; hver `=begin`/`=end`-dokumentasjonsblokk er ett enkelt `type: "block"`-token der `value` er det indre innholdet (markﾃｸrlinjene `=begin`/`=end` er fjernet, tilsvarende fjerningen av `/* */`-skilletegn) og der `loc` spenner fra `=begin`-linjen til og med `=end`-linjen. For Python og Ruby er kommentaruthentingen et eget pass over samme kilde (`tokenize` / `Ripper.lex`); en tokeniseringsfeil pﾃ･ ellers parsbar kode degraderer til en tom kommentarliste i stedet for ﾃ･ feile hele parsingen. TypeScript/JavaScript-skanneren er streng- og malstreng-bevisst, men sporer ikke regulﾃｦrt-uttrykk-literaler, sﾃ･ et kommentarskilletegn inne i en regex-literal er et kjent blindpunkt.

##### Feiloppfﾃｸrsel

`ast()` **kaster** ved feil -- den returnerer aldri `null`:

- **Parsefeil**: filen kan ikke parses som det forespurte sprﾃ･ket. Feilmeldingen inkluderer parserens diagnostikk.
- **Manglende tolk** (kun `python`/`ruby`): ingen egnet tolk ble funnet pﾃ･ `PATH`.
- **Usannsynlig input**: filens filendelse samsvarer ikke med det forespurte sprﾃ･ket (f.eks. kaster `ctx.ast("config.json", "python")` fﾃｸr noen tolk startes).
- **Ingen grunnrevisjon** (kun `{ rev: "base" }`): sjekken kjﾃｸrte uten `--base`, sﾃ･ det finnes ingen basis ﾃ･ parse. Bruk `fileAtBase()` for ﾃ･ oppdage dette som `null` i stedet.
- **Fil ikke til stede ved basisen** (kun `{ rev: "base" }`): stien ble lagt til etter basisen og fantes ikke der.

En kastet feil er isolert til regelen som feiler: andre regler og ADR-er i samme sjekk-kjﾃｸring fortsetter som normalt, og feilen vises som en regelkjﾃｸringsfeil med avslutningskode 2 (til forskjell fra avslutningskode 1 for brudd). Kastetilfellene kan skilles fra hverandre pﾃ･ meldingsteksten, slik at sjekkutdataene skiller "dette miljﾃｸet kan ikke kjﾃｸre denne regelen" fra "denne filen har en syntaksfeil" eller "det finnes ingen grunnrevisjon".

:::caution
Python- og Ruby-regler krever at den tilhﾃｸrende tolken (`python3`/`python`, `ruby`) finnes pﾃ･ `PATH` overalt der `archgate check` kjﾃｸrer -- pﾃ･ hver utviklermaskin **og** i CI. TypeScript- og JavaScript-parsing er innebygd i Archgate og trenger ingen tolk.
:::

## RuleReport

Rapporteringsgrensesnittet for ﾃ･ registrere sjekkresultater. Hver metode aksepterer et detalobjekt som beskriver problemet.

```typescript
interface RuleReport {
  violation(detail: ReportDetail): void;
  warning(detail: ReportDetail): void;
  info(detail: ReportDetail): void;
}
```

#### violation

```typescript
report.violation(detail: ReportDetail): void;
```

Rapporter et regelbrudd. Brudd fﾃ･r sjekken til ﾃ･ feile med avslutningskode 1. Bruk for harde begrensninger som ikke skal merges.

#### warning

```typescript
report.warning(detail: ReportDetail): void;
```

Rapporter en advarsel. Advarsler vises i sjekkutdata, men fﾃ･r ikke sjekken til ﾃ･ feile. Bruk for ikke-blokkerende veiledning.

#### info

```typescript
report.info(detail: ReportDetail): void;
```

Rapporter en informasjonsmelding. Pﾃ･virker ikke avslutningskoden for sjekken. Bruk for forslag eller notater.

### ReportDetail

Detalobjektet som sendes til `violation`, `warning` og `info`.

```typescript
interface ReportDetail {
  message: string;
  file?: string;
  line?: number;
  endLine?: number;
  endColumn?: number;
  fix?: string;
}
```

| Felt        | Type     | Pﾃ･krevd | Beskrivelse                                                                |
| ----------- | -------- | ------- | -------------------------------------------------------------------------- |
| `message`   | `string` | Ja      | Lesbar beskrivelse av problemet                                            |
| `file`      | `string` | Nei     | Filsti der problemet ble funnet                                            |
| `line`      | `number` | Nei     | Startlinjenummer (1-basert)                                                |
| `endLine`   | `number` | Nei     | Sluttlinjenummer (1-basert) -- for presis uthevning av omrﾃ･de i editoren   |
| `endColumn` | `number` | Nei     | Sluttkolonnenummer (0-basert) -- for presis uthevning av omrﾃ･de i editoren |
| `fix`       | `string` | Nei     | Foreslﾃ･tt rettelse eller utbedringstiltak                                  |

Nﾃ･r `endLine` og `endColumn` er oppgitt, kan editorer (VS Code, Cursor) utheve det nﾃｸyaktige uttrykket som bryter regelen, i stedet for hele linjen. Hvis de utelates, utheves hele linjen ved `line`.

## AstNode

Returnert av `ctx.ast()`. Formen er sprﾃ･knativ og bevisst **ikke** enhetlig pﾃ･ tvers av sprﾃ･k -- hvert sprﾃ･k returnerer sitt eget standard AST-vokabular, sﾃ･ en regel som inspiserer Python-kode jobber mot en annen grammatikk enn en som inspiserer TypeScript.

```typescript
type AstLanguage = "typescript" | "javascript" | "python" | "ruby";
type AstNode = Record<string, unknown> | unknown[];
```

Nﾃ･r parset med [`{ comments: true }`](#ast), bﾃｦrer rotnoden ogsﾃ･ en `comments: CommentToken[]`-matrise (alle fire sprﾃ･k; for `ruby` er den en ikke-indeksert egenskap pﾃ･ rot-sexp-matrisen). Den er fravﾃｦrende ellers.

| Sprﾃ･k        | Underliggende parser                                                                                                     | Returnert form                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------------ | ------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `typescript` | [meriyah](https://github.com/meriyah/meriyah), i prosessen, etter at TypeScript er transpilert bort med `Bun.Transpiler` | [ESTree](https://github.com/estree/estree) `Program` med `loc`-posisjonsinfo. Typenivﾃ･-syntaks (`interface`, typealiaser, `export type { ... } from`) fjernes fﾃｸr parsing -- en fil som bare inneholder setninger pﾃ･ typenivﾃ･ parses til en tom `Program`-body. `loc`-posisjoner refererer til det **transpilerte resultatet**, ikke den opprinnelige `.ts`-filen -- fjernede typenivﾃ･-setninger, kommentarer og tomme linjer gjﾃｸr at linjenumrene forskyves, sﾃ･ finn konstruksjonen pﾃ･ nytt i den opprinnelige kildekoden (f.eks. `ctx.readFile()` pluss `indexOf`) fﾃｸr du rapporterer en `line`, eller utelat `line` helt; `loc` er nﾃｸyaktig mot kildekoden bare for `javascript` |
| `javascript` | [meriyah](https://github.com/meriyah/meriyah), i prosessen                                                               | [ESTree](https://github.com/estree/estree) `Program` med `loc`-posisjonsinfo                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| `python`     | Pythons [`ast`-modul](https://docs.python.org/3/library/ast.html) fra standardbiblioteket, via systemtolken              | JSON-serialiserte `ast`-noder: `{ "_type": "Module", "body": [...] }` -- hver node har `_type`, nodens egne felt og posisjonene `lineno` / `col_offset`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| `ruby`       | Rubys [`Ripper`](https://docs.ruby-lang.org/en/master/Ripper.html) fra standardbiblioteket, via systemtolken             | Nestede `Ripper.sexp`-matriser: `["program", [["command", ...]]]` med `[line, column]`-posisjonspar innebygd i token-oppfﾃｸringene                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |

Se [Strukturelle sjekker med ctx.ast()](/guides/writing-rules/#strukturelle-sjekker-med-ctxast) for et komplett eksempel pﾃ･ en regel per sprﾃ･k.

## Inline undertrykkelse

Brudd kan undertrykkes i kildekoden ved hjelp av `archgate-ignore`-kommentarer. Motoren hﾃ･ndterer dette automatisk. Regler trenger ingen spesiell logikk.

```typescript
// archgate-ignore ARCH-006/no-unapproved-deps legacy-avhengighet, migrering planlagt
import chalk from "chalk";
```

En begrunnelse er pﾃ･krevd. Filnivﾃ･-undertrykkelse bruker `archgate-ignore-file`. Stable flere kommentarer for ﾃ･ undertrykke mer enn ﾃｩn regel pﾃ･ samme linje. Se [Opt-out-direktiver](/guides/writing-rules/#opt-out-direktiver) for fullstendige detaljer og egendefinerte direktivmﾃｸnstre.

## ConfiguraĂ§ĂŁo bĂˇsica

Todo arquivo de regras exporta por padrĂŁo um objeto simples tipado com `satisfies RuleSet`. Cada chave no objeto `rules` se torna um ID de regra, e cada regra tem uma `description` e uma funĂ§ĂŁo async `check` que recebe um objeto de contexto.

:::note

## A API de Contexto

O objeto `ctx` passado para toda funĂ§ĂŁo `check` fornece capacidades de leitura de arquivos, busca e relatĂłrio. Aqui estĂˇ uma referĂŞncia detalhada com exemplos.

## VerificaĂ§Ăµes estruturais com ctx.ast()

Regex funciona para padrĂµes de superfĂ­cie, mas falha em perguntas estruturais como "este arquivo contĂ©m apenas re-exports?" ou "isto Ă© uma clĂˇusula `except:` vazia?" -- declaraĂ§Ăµes multilinha, comentĂˇrios e conteĂşdo de strings derrotam a correspondĂŞncia baseada em linhas. `ctx.ast()` faz o parse de um arquivo para uma Ăˇrvore de sintaxe real, permitindo que sua regra verifique a estrutura diretamente.

A Ăˇrvore retornada Ă© nativa da linguagem, nĂŁo unificada entre linguagens:

- **TypeScript / JavaScript** -- um `Program` [ESTree](https://github.com/estree/estree) parseado in-process pelo [meriyah](https://github.com/meriyah/meriyah). O TypeScript Ă© transpilado primeiro, entĂŁo sintaxe exclusivamente de tipos (`interface`, type aliases, `export type { ... } from`) Ă© apagada da Ăˇrvore; um arquivo contendo apenas declaraĂ§Ăµes de nĂ­vel de tipo parseia para um `Program` com `body` vazio.
- **Python** -- a Ăˇrvore do [mĂłdulo `ast`](https://docs.python.org/3/library/ast.html) da biblioteca padrĂŁo serializada em JSON. Cada nĂł Ă© um objeto com um campo `_type` mais os campos do prĂłprio nĂł e as posiĂ§Ăµes `lineno` / `col_offset`.
- **Ruby** -- a saĂ­da de `Ripper.sexp` do [`Ripper`](https://docs.ruby-lang.org/en/master/Ripper.html) da biblioteca padrĂŁo: arrays aninhados como `["program", [["command", ...]]]` com pares de posiĂ§ĂŁo `[linha, coluna]`.

`ctx.ast()` lanĂ§a uma exceĂ§ĂŁo em caso de falha de parse ou de interpretador ausente -- nunca retorna `null`. A exceĂ§ĂŁo fica isolada Ă  regra que falhou e aparece como um erro de execuĂ§ĂŁo de regra (cĂłdigo de saĂ­da 2), entĂŁo um ambiente quebrado se manifesta como uma falha visĂ­vel em vez de um falso sucesso.

:::caution
O parsing de Python e Ruby invoca o interpretador do sistema. O interpretador correspondente (`python3`/`python`, `ruby`) deve estar no `PATH` onde quer que `archgate check` seja executado -- em cada mĂˇquina de desenvolvedor **e** no CI. O parsing de TypeScript e JavaScript Ă© embutido no Archgate e nĂŁo precisa de interpretador.
:::

### TypeScript: sem barrel files

Um barrel file Ă© um `index.ts` cujas declaraĂ§Ăµes de nĂ­vel superior sĂŁo todas re-exports. Com o `Program` ESTree, essa pergunta se torna uma verificaĂ§ĂŁo direta sobre os tipos das declaraĂ§Ăµes, em vez de uma heurĂ­stica de correspondĂŞncia de linhas:

```typescript

      description: "index.ts files must not be pure re-export barrels",
      async check(ctx) {
        const indexFiles = ctx.scopedFiles.filter((f) =>
          f.endsWith("/index.ts")
        );

        const checks = indexFiles.map(async (file) => {
          const program = await ctx.ast(file, "typescript");

          const isBarrel =
            program.body.length > 0 &&
            program.body.every(
              (node) =>
                node.type === "ExportAllDeclaration" ||
                (node.type === "ExportNamedDeclaration" && node.source !== null)
            );

          if (isBarrel) {
            ctx.report.violation({
              message: `Barrel file detected: ${file} contains only re-exports`,
              file,
              fix: "Delete the barrel and import directly from the source modules",
            });
          }
        });

        await Promise.all(checks);
      },
    },
  },
} satisfies RuleSet;
```

Como o TypeScript Ă© transpilado antes do parse, re-exports `export type { ... } from` sĂŁo apagados -- um barrel contendo apenas re-exports de tipos parseia para um `body` vazio, que a guarda `body.length > 0` ignora. Se vocĂŞ tambĂ©m precisa sinalizar barrels exclusivamente de tipos, combine a verificaĂ§ĂŁo de AST com uma verificaĂ§ĂŁo textual. A mesma transpilaĂ§ĂŁo tambĂ©m desloca posiĂ§Ăµes: os nĂşmeros de linha em `loc` referem-se ao texto transpilado, nĂŁo ao seu arquivo `.ts` original, entĂŁo uma regra `"typescript"` precisa relocalizar a construĂ§ĂŁo no cĂłdigo-fonte original (por exemplo com `ctx.readFile()` e `indexOf`) antes de reportar um `line` -- ou omitir `line`.

### JavaScript: sem require() em mĂłdulos ES

Percorra a Ăˇrvore ESTree em busca de nĂłs `CallExpression` cujo callee Ă© o identificador `require`. Um percurso recursivo sobre valores de objetos e arrays cobre todos os tipos de nĂł sem enumerĂˇ-los:

### Python: sem clĂˇusulas except vazias

No mĂłdulo `ast` do Python, uma clĂˇusula `except:` Ă© um nĂł `ExceptHandler` cujo campo `type` guarda a expressĂŁo da exceĂ§ĂŁo capturada -- um `except:` vazio tem `"type": null`. Percorra a Ăˇrvore JSON em busca desse formato:

### Ruby: sem chamadas puts

O `Ripper.sexp` representa `puts "hello"` como `["command", ["@ident", "puts", [1, 0]], [...args]]` -- o par `[linha, coluna]` fica dentro do token `@ident`. Percorra os arrays aninhados em busca desse formato:

```typescript
/// <reference path="../rules.d.ts" />

  if (
    kind === "command" &&
    Array.isArray(first) &&
    first[0] === "@ident" &&
    first[1] === "puts"
  ) {
    const [line] = first[2] as [number, number];
    lines.push(line);
  }
  for (const child of node) findPutsCalls(child, lines);
}

  rules: {
    "no-puts": {
      description: "Ruby code must use the application logger, not puts",
      async check(ctx) {
        const files = await ctx.glob("app/**/*.rb");

        const checks = files.map(async (file) => {
          const sexp = await ctx.ast(file, "ruby");
          const lines: number[] = [];
          findPutsCalls(sexp, lines);
          for (const line of lines) {
            ctx.report.violation({
              message: "puts writes to stdout directly",
              file,
              line,
              fix: "Replace puts with logger.info",
            });
          }
        });

        await Promise.all(checks);
      },
    },
  },
} satisfies RuleSet;
```

O Ripper usa um formato de nĂł diferente para a forma com parĂŞnteses -- `puts("hello")` aparece sob um par `method_add_arg` / `fcall`, em vez de `command` -- entĂŁo uma regra de produĂ§ĂŁo casaria o token `fcall` da mesma maneira.

### Comparando com a revisĂŁo base

Algumas regras nĂŁo se importam com o estado atual de um arquivo, mas com **o que mudou**. ```typescript

```typescript
        for (const file of changed) {
          // Sem base para comparar (ex.: um arquivo adicionado) -- pular.
          if ((await ctx.fileAtBase(file)) === null) continue;

          const before = await ctx.ast(file, "python", { rev: "base" });
          const after = await ctx.ast(file, "python");
          if (JSON.stringify(before) !== JSON.stringify(after)) {
            ctx.report.violation({
              message: `${file} changed behavior -- bump the version`,
              file,
            });
          }
        }
      },
    },
  },
} satisfies RuleSet;
```

Isso roda com `archgate check --base <ref>`; sem uma base, `ctx.changedFiles` fica vazio e a regra Ă© um no-op. Como o parse da base passa por `ctx.ast()` e seu acesso ao git Ă© embutido no Archgate, a regra nunca executa `git` ou um interpretador por conta prĂłpria -- o que o [sandbox de regras](/guides/security/) bloquearia de qualquer forma.

## NĂ­veis de severidade

Cada regra pode definir uma severidade padrĂŁo em sua configuraĂ§ĂŁo. A severidade determina como as violaĂ§Ăµes sĂŁo tratadas:

| Severidade | CĂłdigo de saĂ­da | Comportamento                               |
| ---------- | --------------- | ------------------------------------------- |
| `error`    | 1               | Bloqueia CI, deve ser corrigido             |
| `warning`  | 0               | Registrado, mas nĂŁo bloqueia                |
| `info`     | 0               | Informacional, registrado para visibilidade |

Defina a severidade na definiĂ§ĂŁo da regra:

```typescript

    "my-rule": {
      description: "...",

```

Se `severity` for omitido, o padrĂŁo Ă© `error`.

VocĂŞ tambĂ©m pode reportar com diferentes severidades dentro da mesma regra usando `ctx.report.violation()`, `ctx.report.warning()` e `ctx.report.info()` diretamente.

name: project-ci-workflow-gotchas
description: GitHub Actions gotchas outside the release pipeline (token permissions, config namespaces, shell encoding)
metadata:
  type: project
---

- **`GITHUB_TOKEN`-authored pushes do NOT trigger downstream workflows.** A workflow pushing commits/PRs with `github.token`/`secrets.GITHUB_TOKEN` suppresses the resulting `push`/`pull_request` events (anti-recursion), so required PR checks never run. Fix: author such pushes with a GitHub App installation token (`actions/create-github-app-token`) on both `actions/checkout` and the pushing step.
- **`secrets.*` and `vars.*` are separate, non-overlapping namespaces** — configuring a value as one does not make it readable via the other. `release.yml`'s PostHog annotation step once read `vars.POSTHOG_PROJECT_ID` when the value only existed as a **secret**, silently no-opping for weeks behind `continue-on-error: true` + a low-visibility `::notice::`. Confirm the actual location with `gh secret list`/`gh variable list` before writing a reference. For any `continue-on-error` step with a skip-on-missing-config guard, use `::warning::` (not `::notice::`) so misconfiguration is visible in the Actions UI.
- **zizmor suppressions must be inline comments, not zizmor.yml line-number ignores.** A `rules.<rule>.ignore: [file.yml:172]` entry silently stops matching when the line drifts (this resurfaced the SLSA `unpinned-uses` alert). Use `# zizmor: ignore[rule]` at the end of the flagged line instead; zizmor.yml keeps only `rules: {}` plus the justification index. Verify locally with `uvx zizmor --config zizmor.yml .github/workflows/`.
- **`actions/create-github-app-token` needs `permission-*` inputs** (e.g. `permission-contents: write`) — without them the token inherits every permission of the app installation (zizmor `github-app` finding). **Requested scopes MUST be a subset of the installation's grants or token creation 422s** ("The permissions requested are not granted to this installation") — requesting `permission-issues: write` broke the Release workflow on main because archgatebot is only granted contents+pull_requests+metadata. Check grants with `gh api orgs/archgate/installations`. Scope per workflow: update-lock/update-llms → `permission-contents: write`; release.yml → contents+pull-requests write (app-token PR comments are covered by pull_requests, not issues).
- **Publish workflows must not enable action caches** (setup-uv `enable-cache: false`, setup-ruby `bundler-cache: false` + explicit uncached `bundle install`) — zizmor `cache-poisoning`: a poisoned cache restored during publish can contaminate the released artifact.
- **Scorecard alerts only close on the next scheduled scorecard.yml run** (weekly); zizmor alerts close on the next push-to-main scan. Don't expect immediate alert closure after merging fixes.
- **jq on Windows Git Bash emits CRLF line endings.** `jq -r` output carries a trailing `\r` after command-substitution newline-stripping (and mid-list entries too). Pipe through `tr -d '\r'`. Broke `install.sh`'s `resolve_version` release-walk (skipped every tag but the last).
